7 Data Structure and Format

7.1 The Shape of Nested Data

Before fitting any model, you need your data in the right shape. Nested data analysis requires long format: one row per observation at the lowest level of the hierarchy, with identifier variables that track higher-level units.

This sounds simple. In practice, it trips up a surprising number of researchers.

7.2 Long Format vs. Wide Format

Most people who have worked with repeated measures data in SPSS or Excel have encountered data in wide format: one row per person, with multiple columns for each time point or condition.

Code 
# Wide format example
df_wide <- tibble(
  person_id = 1:6,
  sex       = c("F", "M", "F", "F", "M", "M"),
  age       = c(23, 31, 27, 25, 29, 34),
  mood_day1 = c(65, 72, 58, 70, 63, 55),
  mood_day2 = c(68, 69, 61, 74, 60, 58),
  mood_day3 = c(71, 67, 64, 72, 65, 62),
  mood_day4 = c(66, 74, 59, 75, 61, 57),
  mood_day5 = c(70, 70, 63, 78, 64, 60)
)

knitr::kable(df_wide, caption = "Wide format: 6 people × 5 days = 6 rows")

This format feels intuitive — it keeps all information about one person on one row. But it is not suitable for multilevel modeling. Why?

Because the model needs to see the nesting explicitly. It needs to know that mood_day1, mood_day2, and mood_day3 belong to the same person — that they are three realizations of the same level-1 unit nested inside one level-2 unit.

Long format makes the nesting explicit:

Code 
df_long <- df_wide |>
  pivot_longer(
    cols      = starts_with("mood_"),
    names_to  = "day",
    values_to = "mood"
  ) |>
  mutate(
    day = as.integer(str_remove(day, "mood_day"))
  ) |>
  select(person_id, sex, age, day, mood)

7.3 Converting Wide to Long in R

Code 
# The pivot_longer() function does the conversion
df_long <- df_wide |>
  pivot_longer(
    cols      = starts_with("mood_"),   # which columns to collapse
    names_to  = "day",                  # new column for what the columns were about
    values_to = "mood"                  # new column for the values
  ) |>
  mutate(
    day = as.integer(str_remove(day, "mood_day"))  # clean up the day variable
  )

7.4 The Essential Identifier Variables

Every nested dataset needs at least two types of identifier variables:

  1. Level-2 identifier: A unique code for each higher-level unit (person, firm, classroom, country). This is what tells the model which observations belong together.
  2. Level-1 identifier or index: A variable that distinguishes observations within the same level-2 unit (day number, condition, trial number).

In three-level data, you need identifiers at all three levels.

7.5 Level-1 vs. Level-2 Variables

A common source of confusion: variables that describe the person versus variables that describe the observation.

Code 
df_long |>
  head(10) |>
  knitr::kable(caption = "Identifying level in the dataset")
Identifying level in the dataset
person_id sex age day mood
1 F 23 1 65
1 F 23 2 68
1 F 23 3 71
1 F 23 4 66
1 F 23 5 70
2 M 31 1 72
2 M 31 2 69
2 M 31 3 67
2 M 31 4 74
2 M 31 5 70

In this dataset: - person_id, sex, age are level-2 variables — they describe the person and are constant across all of that person’s rows - day and mood are level-1 variables — they vary across observations within a person

This distinction matters enormously for model building (centering, interpretation of effects) and will come up repeatedly in Module 2.

7.6 A Note on Balanced vs. Unbalanced Data

Balanced data means every level-2 unit has the same number of level-1 observations. All 30 people have exactly 5 days of data. All 25 classrooms have exactly 20 students.

Unbalanced data means the number of observations varies across level-2 units. Some people completed 7 days of the diary; others dropped out after 2. Some classrooms have 15 students; others have 28.

Standard ANOVA-based approaches for repeated measures struggle with unbalanced data — they were designed for balanced designs. Mixed-effects models handle unbalanced data gracefully, because they use the information available for each unit without requiring complete cases.

Code 
# Show what unbalanced data looks like
df_unbalanced <- df_long |>
  # randomly drop some observations to simulate dropout
  filter(!(person_id %in% c(1, 4) & day >= 4),
         !(person_id == 6 & day >= 3))

df_unbalanced |>
  group_by(person_id) |>
  summarise(n_obs = n(), days_observed = paste(day, collapse = ", ")) |>
  knitr::kable(caption = "Unbalanced data: some people have fewer observations")

7.7 Diagnosing Your Data Structure

Before fitting any model, it is good practice to run a quick structure check:

Code 
# How many observations per person?
df_long |>
  count(person_id, name = "n_obs") |>
  summary()
   person_id        n_obs  
 Min.   :1.00   Min.   :5  
 1st Qu.:2.25   1st Qu.:5  
 Median :3.50   Median :5  
 Mean   :3.50   Mean   :5  
 3rd Qu.:4.75   3rd Qu.:5  
 Max.   :6.00   Max.   :5
Code 
# Is the data truly in long format? Check that level-2 variables don't vary within person
df_long |>
  group_by(person_id) |>
  summarise(
    n_unique_sex = n_distinct(sex),   # should be 1
    n_unique_age = n_distinct(age)    # should be 1
  ) |>
  filter(n_unique_sex > 1 | n_unique_age > 1)  # should return 0 rows
person_id n_unique_sex n_unique_age

Zero rows returned means no person has inconsistent sex or age coding across their observations. Good.

Code 
# How many level-2 units?
n_distinct(df_long$person_id)
[1] 6
Code 
# What is the range of level-1 observations per unit?
df_long |>
  count(person_id) |>
  pull(n) |>
  range()
[1] 5 5

This takes two minutes and saves hours of debugging later.
